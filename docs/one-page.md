## 1. 프로젝트 개요

### 1.1 한 줄 정의

개인 일기를 **웹에서 작성·조회**하고, **제목 검색**·**날짜 필터(미니 달력)**로 찾을 수 있는 단일 사용자용 애플리케이션.

### 1.2 목표

- **TipTap** 기반 **리치 편집**, **이미지**(드래그·드롭·붙여넣기)까지 자연스러운 작성 경험
- 데이터는 **본인만** 접근 (Supabase Auth + RLS)
- **감정·날씨 이모지**로 하루 감각을 빠르게 기록·표시
- **한국 시간** 기준으로 일기 날짜·표시 일관성 유지

### 1.3 비목표 (초기 버전)

- 다중 사용자·소셜·공개 피드
- 앱스토어 네이티브 앱
- 실시간 협업

### 1.4 기술 가정

- 앱: **Next.js (App Router)**, **React**, **Tailwind**, **shadcn/ui**
- 데이터·파일·인증: **Supabase** (PostgreSQL, Auth, Storage)
- 서버 변이: **Server Actions** 우선 (필요 시 Route Handler 보조)

## 2. 시스템 구조

### 2.1 논리 구성

```text
[브라우저]
    │
    ▼
[Next.js 앱]
    ├── Server Actions: 일기 CRUD, 이미지 업로드 메타·권한 검증, 검색 쿼리
    ├── 클라이언트: TipTap, 사이드바 미니 달력, 목록·상세·편집 UI
    └── 국제화(선택): ko / en UI 문자열·날짜 포맷
    │
    ▼
[Supabase]
    ├── PostgreSQL (일기·메타·휴지통)
    ├── Auth
    └── Storage (에디터 삽입 이미지)
```

### 2.2 배포·접속

- **인터넷 배포** + **로그인** 필수 전제 유지.
- 호스팅·도메인 세부는 [§8](#8-남은-결정질문).

### 2.3 보안·인증

- **Supabase Auth** + **RLS** (`auth.uid()` 기준 본인 데이터만).

### 2.4 라우팅·페이지 책임 (사전 결정과 동일)

- **`/`**: 전체 목록(사이드바 달력으로 날짜 필터 가능)
- **`/new`**: 신규 작성
- **`/diary/[id]`**: 상세
- **`/diary/[id]/edit`**: 편집

## 4. UI/UX 설계

### 4.1 정보 구조·레이아웃

- **좌측 사이드바(또는 모바일에서 Drawer)**:
  - **미니 달력**: 날짜 클릭 시 목록 **해당 일자로 필터** (전체 보기 토글 명확히)
  - **라이트/다크 전환** 버튼
  - (선택) 검색(제목), 새 글 `/new` 링크
- **메인**: `/` 목록, `/diary/[id]` 읽기, `/new`·`/edit` 작성/편집
- **감정·날씨 이모지**: 작성 폼에서 선택 → 카드/상세에 동일 표시

### 4.2 에디터 UX (TipTap)

- 툴바·단축키·이미지 확장: **드래그 앤 드롭**, **클립보드 붙여넣기** 처리 후 Storage 업로드 → 문서에 URL 삽입.
- 날짜 필드: **KST** 기준 달력·표시 (`date-fns-tz` + `Asia/Seoul` 또는 동등 처리).

### 4.3 시각 방향

- **라이트 기본**, 다크는 사이드바 토글.
- **shadcn/ui** 컴포넌트 우선으로 톤 통일.

### 4.4 접근성

- shadcn·Radix 기본 + 달력·에디터 포커스 이동 점검.

### 4.5 저장 형식 (TipTap 확정에 따른 정리)

- 편집기는 **TipTap** 고정 → 저장은 일반적으로 **JSON(ProseMirror 문서)** 또는 **HTML** 중 하나(또는 병행: JSON 주 저장 + 검색용 HTML). **Markdown 내보내기(F-22)**는 별도 변환 파이프라인으로 후순위 처리.
- 이미지는 본문에 **Storage 공개/서명 URL** 삽입; DB에는 본문 blob + `user_id`로 RLS.

### 4.6 국제화(i18n) — 한국어·영어

- 메시지: `messages/ko.json`, `messages/en.json`; **`next-intl`** 등 후보 유지.
- **일기 날짜·달력**은 비즈니스 규칙상 **KST 고정**(UI 언어와 무관).
- 라우팅: [사전 결정 라우팅](#라우팅-구조-url)과 병행하려면 `app/[locale]/page.tsx` 등으로 감싸 동일 세그먼트를 유지하거나, locale은 쿠키만 쓰고 URL은 위 표와 동일하게 유지할 수 있음(§8).

## 5. 데이터베이스

### 5.1 플랫폼

- **Supabase PostgreSQL**, **Storage**, **Auth**, **RLS**.

### 5.2 엔티티 (초안)

**일기 (`entries`)**

| 필드 | 타입(개념) | 설명 |
|------|------------|------|
| `id` | uuid | PK |
| `user_id` | uuid | FK → `auth.users` |
| `entry_date` | date | **KST 기준** 사용자가 지정한 일기 날짜 |
| `title` | text | 필수; **제목 검색** 대상 |
| `body` | text / jsonb | TipTap 문서(JSON) 또는 HTML 등 구현 선택 |
| `mood_emoji` | text, nullable | 감정 이모지(유니코드 문자열) |
| `weather_emoji` | text, nullable | 날씨 이모지 |
| `created_at` | timestamptz | UTC 저장 권장, 표시는 KST 변환 |
| `updated_at` | timestamptz | |
| `deleted_at` | timestamptz, nullable | 휴지통 |

**스토리지 메타 (선택)**

- 큰 규모·정리용으로 `entry_images(id, entry_id, storage_path, …)` 테이블을 둘지, 본문 URL만으로 갈지 구현 단계에서 선택.

### 5.3 검색·필터

- **제목**: `ILIKE` 또는 `pg_trgm`(선택)으로 부분 일치.
- **날짜 필터**: 사이드바 달력 → `entry_date = 선택일` (KST로 맞춘 date).

### 5.4 백업·내보내기

- Supabase 프로젝트 백업 + (선택) 앱 내 Markdown/ZIP 내보내기.

## 6. 개발 순서

1. Supabase **Auth**·**RLS**·환경 변수; `entries` 스키마(휴지통·이모지·`entry_date`)
2. **Server Actions**: CRUD, 제목 검색, 날짜 필터 쿼리
3. 앱 레이아웃: **반응형 사이드바** + **미니 달력** + **테마 토글**
4. 라우트: `/`, `/new`, `/diary/[id]`, `/diary/[id]/edit`
5. **TipTap** + **이미지**(Storage 업로드 액션, 드래그·붙여넣기)
6. 감정·날씨 이모지 UI·저장
7. **i18n** 골격(선택과 동시에 가도 됨)
8. 휴지통·복구·영구 삭제
9. 내보내기·백업 UX(후순위)
